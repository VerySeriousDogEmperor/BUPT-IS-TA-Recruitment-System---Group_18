## 部署相关问题

### 问题 1.1：Tomcat 部署失败

**问题描述**：
- 尝试使用 Tomcat 11 部署 WAR 文件
- IntelliJ IDEA 的 Tomcat 配置中 `deployIgnore` 正则表达式阻止了 Web 应用部署
- Java 版本不匹配（代码用 Java 22 编译，Tomcat 运行在 Java 17）

**错误信息**：
```
Application not deployed
ClassCastException: jakarta.servlet vs javax.servlet
```

**排查过程**：
1. 检查 Tomcat 配置文件
2. 发现 `deployIgnore` 配置问题
3. 检查 Java 版本兼容性
4. 发现 servlet-api.jar 冲突

**解决方案**：
- **最终决策**：放弃 Tomcat，改用嵌入式 HTTP 服务器
- 使用 JDK 内置的 `HttpServer`
- 实现适配器模式桥接到 Servlet API

**经验教训**：
- Tomcat 部署配置复杂，不适合快速原型开发
- 嵌入式服务器更简单，无需外部依赖
- 注意 Jakarta EE 和 Java EE 的命名空间差异（jakarta.servlet vs javax.servlet）

---

### 问题 1.2：端口冲突

**问题描述**：
- 初始使用 8080 端口，与其他服务冲突
- 启动时报错 `Address already in use`

**错误信息**：
```
java.net.BindException: Address already in use: bind
```

**解决方案**：
- 修改端口为 9191
- 在 `EmbeddedServer.java` 中配置：
  ```java
  private static final int PORT = 9191;
  ```

**经验教训**：
- 避免使用常用端口（8080, 8888 等）
- 提供端口配置选项

## 编译相关问题

### 问题 2.1：缺少 Jakarta Servlet API JAR 包

**问题描述**：
- 编译时报错找不到 `jakarta.servlet.http` 包
- `lib` 目录只有 `gson-2.13.2.jar`

**错误信息**：
```
错误: 程序包jakarta.servlet.http不存在
import jakarta.servlet.http.HttpServletResponse;
                           ^
```

**排查过程**：
1. 检查 `lib` 目录内容
2. 确认缺少 servlet-api JAR 包
3. 搜索 Maven Central 仓库

**解决方案**：
1. 下载 `jakarta.servlet-api-6.0.0.jar`
   ```bash
   Invoke-WebRequest -Uri "https://repo1.maven.org/maven2/jakarta/servlet/jakarta.servlet-api/6.0.0/jakarta.servlet-api-6.0.0.jar" -OutFile "lib/jakarta.servlet-api-6.0.0.jar"
   ```

2. 或手动下载：
   - 访问：https://repo1.maven.org/maven2/jakarta/servlet/jakarta.servlet-api/6.0.0/
   - 下载 `jakarta.servlet-api-6.0.0.jar`
   - 放到 `lib` 目录

**经验教训**：
- 项目依赖要完整记录
- 提供依赖下载脚本
- 考虑使用 Maven/Gradle 管理依赖

### 问题 2.2：编译顺序导致的循环依赖

**问题描述**：
- 直接编译所有文件时出现找不到类的错误
- `DataInitializer` 依赖 `Repository`，`Repository` 依赖 `JsonFileUtil`

**错误信息**：
```
错误: 找不到符号
  符号:   类 StudentRepository
```

**解决方案**：
按正确顺序编译：
```bash
# 1. Domain 层（无依赖）
javac -encoding UTF-8 -cp "lib/*" -d out src/com/bupt/ta/shared/domain/*.java

# 2. 基础 Util 类（只依赖 Domain）
javac -encoding UTF-8 -cp "lib/*;out" -d out src/com/bupt/ta/shared/util/JsonFileUtil.java

# 3. Infrastructure 层（依赖 Domain + Util）
javac -encoding UTF-8 -cp "lib/*;out" -d out src/com/bupt/ta/shared/infrastructure/*.java

# 4. 其他 Util 类（依赖 Infrastructure）
javac -encoding UTF-8 -cp "lib/*;out" -d out src/com/bupt/ta/shared/util/*.java

# 5. Interfaces 层
javac -encoding UTF-8 -cp "lib/*;out" -d out src/com/bupt/ta/shared/interfaces/*.java
```

**经验教训**：
- 设计时避免循环依赖
- 提供编译脚本
- 考虑使用构建工具（Maven/Gradle）

### 问题 2.3：编译警告 - unchecked operations

**问题描述**：
- 编译时出现大量 unchecked 警告
- 主要是泛型类型转换问题

**警告信息**：
```
注: 某些输入文件使用了未经检查或不安全的操作。
注: 有关详细信息, 请使用 -Xlint:unchecked 重新编译。
```

**原因分析**：
- `readRequestBody()` 方法使用了原始类型的 `Class`
- JSON 反序列化时类型擦除

**解决方案**：
- 警告不影响运行，可以忽略
- 如需消除警告，使用 `@SuppressWarnings("unchecked")`

**经验教训**：
- 泛型使用要规范
- 区分警告和错误的严重性

## 运行时问题

### 问题 3.1：登录 API 返回 404

**问题描述**：
- 前端调用 `/api/auth/login` 返回 404
- 服务器已启动，但接口不存在

**错误信息**：
```javascript
接口不存在
```

**排查过程**：
1. 检查 `EmbeddedServer.java` 路由注册
2. 检查 `HttpServletRequestAdapter.getPathInfo()` 实现
3. 发现路径解析错误

**问题原因**：
- `getPathInfo()` 没有正确去除 context path
- 路径匹配失败

**解决方案**：
修改 `HttpServletRequestAdapter.java`：
```java
@Override
public String getPathInfo() {
    String path = exchange.getRequestURI().getPath();
    // 去除 context path
    if (path.startsWith(contextPath)) {
        path = path.substring(contextPath.length());
    }
    return path.isEmpty() ? "/" : path;
}
```

**经验教训**：
- 适配器实现要完整测试
- 路径处理要考虑各种情况

### 问题 3.2：登录后底部按钮未更新

**问题描述**：
- 学生登录后，底部仍显示"Student Login"和"Staff Entrance"
- 应该显示"My Dashboard"和"Logout"

**原因分析**：
- 前端没有检查登录状态
- 页面加载时未更新 UI

**解决方案**：
修改 `web/index.html`：
```javascript
// 页面加载时检查登录状态
window.addEventListener('DOMContentLoaded', () => {
    const isAuthenticated = localStorage.getItem('isAuthenticated') === 'true';
    const userRole = localStorage.getItem('userRole');
    
    if (isAuthenticated) {
        document.getElementById('guestButtons').style.display = 'none';
        document.getElementById('userButtons').style.display = 'flex';
    }
});
```

**经验教训**：
- 前后端状态要同步
- 登录状态要持久化

### 问题 3.3：Session 管理问题

**问题描述**：
- 多次请求 Session 丢失
- 用户需要重复登录

**原因分析**：
- `HttpSessionAdapter` 实现不完整
- Session 存储在内存中，服务器重启后丢失

**解决方案**：
1. 完善 `HttpSessionAdapter` 实现
2. 使用 `ConcurrentHashMap` 存储 Session
3. 设置合理的 Session 超时时间

**经验教训**：
- Session 管理要考虑并发
- 生产环境考虑使用 Redis 等持久化方案

## 架构设计问题

### 问题 4.1：DDD 分层不清晰

**问题描述**：
- 初期代码混乱，业务逻辑和数据访问混在一起
- 难以维护和扩展

**解决方案**：
采用 DDD 四层架构：
```
Interfaces 层（API 接口）
    ↓
Application 层（业务逻辑）
    ↓
Domain 层（领域模型）
    ↓
Infrastructure 层（数据访问）
```

**实施细节**：
- Interfaces：Servlet 处理 HTTP 请求
- Application：协调多个 Repository
- Domain：实体类，包含业务规则
- Infrastructure：Repository 封装数据访问

**经验教训**：
- 架构设计要提前规划
- 分层要清晰，职责要明确

---

### 问题 4.2：适配器模式实现复杂

**问题描述**：
- 需要实现大量 Servlet API 接口方法
- 很多方法用不到，但必须实现

**解决方案**：
- 只实现必要的方法
- 其他方法提供空实现或抛出 `UnsupportedOperationException`
- 重点实现：
  - `getPathInfo()`
  - `getParameter()`
  - `getSession()`
  - `getWriter()`
  - `setStatus()`

**经验教训**：
- 适配器模式要根据实际需求实现
- 不必追求完美，够用即可

## API 接口问题

### 问题 6.1：SessionUtil 类型转换问题

**问题描述**：
- `getCurrentUser()` 返回 `Object` 类型
- MO Servlet 中需要强制类型转换为 `User`
- 编译报错：`不兼容的类型: Object无法转换为User`

**错误代码**：
```java
User currentUser = SessionUtil.getCurrentUser(request); // 编译错误
```

**解决方案**：
添加类型安全的方法：
```java
// 获取学生对象
public static Student getCurrentStudent(HttpServletRequest request) {
    Object user = getCurrentUser(request);
    if (user instanceof Student) {
        return (Student) user;
    }
    return null;
}

// 获取 MO/Admin 对象
public static User getCurrentMOUser(HttpServletRequest request) {
    Object user = getCurrentUser(request);
    if (user instanceof User) {
        return (User) user;
    }
    return null;
}
```

**经验教训**：
- 避免返回 Object 类型
- 提供类型安全的 API
- 使用泛型或重载方法

---

### 问题 6.2：权限验证不统一

**问题描述**：
- 每个 Servlet 都要重复写权限验证代码
- 代码冗余，容易遗漏

**解决方案**：
在 `BaseServlet` 中提供统一的权限验证方法：
```java
protected boolean checkRole(HttpServletRequest request, HttpServletResponse response, String requiredRole) throws IOException {
    User user = SessionUtil.getCurrentMOUser(request);
    if (user == null || !requiredRole.equals(user.getRole())) {
        ResponseUtil.sendError(response, 401, "未授权");
        return false;
    }
    return true;
}
```

**经验教训**：
- 提取公共逻辑到基类
- 使用拦截器或过滤器统一处理

---

## 前端集成问题

### 问题 7.1：CORS 跨域问题

**问题描述**：
- 前端开发时使用不同端口
- 浏览器报 CORS 错误

**解决方案**：
在 `EmbeddedServer` 中添加 CORS 支持：
```java
exchange.getResponseHeaders().add("Access-Control-Allow-Origin", "*");
exchange.getResponseHeaders().add("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, OPTIONS");
exchange.getResponseHeaders().add("Access-Control-Allow-Headers", "Content-Type");
```

**经验教训**：
- 开发环境要配置 CORS
- 生产环境要限制允许的域名

### 问题 7.2：静态资源路径问题

**问题描述**：
- 前端引用的 CSS/JS 路径不正确
- 404 错误

**解决方案**：
在 `EmbeddedServer` 中添加静态资源处理：
```java
server.createContext("/", exchange -> {
    String path = exchange.getRequestURI().getPath();
    if (path.equals("/")) {
        path = "/index.html";
    }
    
    File file = new File("web" + path);
    if (file.exists() && file.isFile()) {
        // 返回文件内容
    } else {
        // 返回 404
    }
});
```

**经验教训**：
- 静态资源要单独处理
- 设置正确的 Content-Type

## 安全问题

### 问题 9.1：密码明文存储

**问题描述**：
- 密码以明文形式存储在 JSON 文件中
- 存在安全风险

**当前状态**：
- 开发环境使用明文密码
- 已在文档中标注需要改进

**建议方案**：
```java
// 使用 BCrypt 加密
String hashedPassword = BCrypt.hashpw(password, BCrypt.gensalt());

// 验证密码
boolean isValid = BCrypt.checkpw(inputPassword, hashedPassword);
```

**经验教训**：
- 密码必须加密存储
- 使用成熟的加密算法

---

### 问题 9.2：SQL 注入风险

**问题描述**：
- 虽然使用 JSON 存储，但字符串拼接仍有风险

**解决方案**：
- 使用参数化查询（如果迁移到数据库）
- 输入验证和过滤
- 使用 ORM 框架

**经验教训**：
- 永远不要信任用户输入
- 使用白名单验证

---

### 问题 9.3：XSS 攻击风险

**问题描述**：
- 用户输入直接显示在页面上
- 可能被注入恶意脚本

**解决方案**：
- 前端使用 `textContent` 而不是 `innerHTML`
- 后端对输入进行 HTML 转义
- 设置 CSP 头

**经验教训**：
- 输出要转义
- 使用安全的 API
